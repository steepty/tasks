# 3) Схема-описание работы сервисов

Этот пункт будет только на первых этапах, чтобы просто проследить ход мыслей разработчика. Для каждого из возможных сервисов создается первичная документация, ожидается что-то подобное

Это пример
# OrderService

Сервис для приёма заявок от заказчиков. Принимает заявления, проверяет их корректность и сохраняет в хранилище для дальнейшей обработки.

## API

### POST /api/v1/orders

Приём новой заявки.

**Запрос:**
```json
{  
  "customer_id": "cust-123",  
  "order_type": "installation",  
  "details": "Установить камеру в офисе",  
  "priority": "high",  
  "timestamp": 1733567890  
}
```

|Поле|Тип|Описание|
|---|---|---|
|`customer_id`|string|Идентификатор заказчика|
|`order_type`|string|Тип заявки (например, установка, ремонт, поддержка)|
|`details`|string|Описание заявки|
|`priority`|string|Приоритет заявки (`low`, `medium`, `high`)|
|`timestamp`|int|Unix timestamp создания заявки|

**Ответ (200 OK):**
```json
{  
  "status": "accepted",  
  "order_id": "order-456"  
}
```


---

### GET /api/v1/orders/{order_id}

Получение информации по заявке.

**Ответ:**
```json
{  
  "order_id": "order-456",  
  "customer_id": "cust-123",  
  "order_type": "installation",  
  "details": "Установить камеру в офисе",  
  "priority": "high",  
  "status": "pending",  
  "created_at": 1733567890  
}
```


---

### GET /api/v1/stats

Статистика работы сервиса.

**Ответ:**
```json
{  
  "total_orders": 452,  
  "pending_orders": 12,  
  "processed_orders": 440,  
  "errors_count": 3,  
  "uptime": "12h15m30s"  
}
```


---

## Как работает

Клиент → POST /orders → Валидатор → Хранилище

1. Клиент отправляет заявку на `/api/v1/orders`
2. Сервис проверяет корректность данных
3. Сохраняет заявку в хранилище
4. Доступно получение статуса и статистики работы через API

  

Описание максимально подробное , **_не обязательно делать также подробно_**  и тратить час или больше времени , чтобы описать каждую ручку, какие именно 20 полей ей приходят и что именно она отдает, важна просто схема и понимание деятельности. Или просто 1-2-3 ручки сделать качественно, или сделать по немного все , чтобы было понятно, что разработчик понимает как это работает

+  Основной критерий этого пункта **_Разработчик понимает как работают его сервисы, какие данные приходят, какие есть ограничения и особенности взаимодействия с его сервисами. Понимает также, как именно проектировать эндпоинты, чтобы было удобно работать как  пользователям, так и самому разработчику_**

 
# 4) Выбор стека технологий для реализации сервисов

Этот пункт тоже будет только на первых этапах, больше для понимания и проверки самостоятельной работы

Пункт подразумевает самостоятельную работу и последующий процесс задания появившихся вопросов по этому поводу. В рамках этого пункта стоит использовать ещё и имеющиеся нейросети/ai помощники для изучения материалов и поиска информации.

На рынке го библиотек имеется нескончаемое количество разнообразных решений, важно научиться выбирать нужное.

В рамках данного проекта нужно решить сразу несколько вопросов выбора библиотек:
+ Что использовать для реализации взаимодействия с сервисом? Если это будет HTTP/gRPC/веб сокет, то что взять
+ Какую библиотеку выбрать для работы с БД?
+ Каким образом конфиг должен загружаться в приложение, если необходимо читать его из файла
+ Каким образом будут писаться логи, и где они будут храниться

+ Основной критерий этого пункта **_Разработчик понимает какие библиотеки за что отвечают, для чего используется , самый важный фактор , что разработчик может объяснить выбор и почему в каком-то случае достаточно функционала стандартной библиотеки, а в каком-то нет. Это не требование получить список , что для бд мы берем pgx , а для логов zerolog, а понимание , где хватает stdlib, а где нет, и как это усилить. Сюда же стоит отнести осознанность в использовании фреймворков, чтобы они не были критической точкой проекта_**
