
## №1 Биржа фриланс заказов (или любых других)
+ Примерные кейсы
	+ Заказчит публикует задачу и платит за ней, получает отклики от исполнителей
	+ Исполнитель получает заказы , деньги после выполнения
	+ Система держит деньги в заморозке (или как угодно иначе) пока работа не завершена , система выступает своеобразным гарантом честности операции
 ---
+ Основные сложности 
	+ Задача должна быть в работе у одного, нельзя допустить дублирования для разных исполнителей
	+ Механизм получения оплаты от исполнителя и заморозки до момента выполнения задачи или отказа от выполнения и возврата денег
	+ Работа с конечными автоматами
	+ Интеграция веб-сокетов для риалтайм статусов работы
---
+ Примерный стек по технологиям
	+ SQL БД 
	+ Очередь (сразу скажу здесь и нужна, и нет , нужно выбрать и обосновать )
	+ Веб-сокеты (горилла, тут особо без альтернатив)
	+ jwt (обосновать почему и для чего)
	+ докер компоуз 
	+ Требования
	+ Реализовать работоспособный вариант
	+ Реализовать все требования по обсервабилити
	+ Проведение тестирования всем корнеркейсов, чтобы готовый вариант гарантированно не допускал дублирования задач между исполнителями, а все статусы гарантированно быстро обновлялись и попадали к получателю




## №2 Файловое хранилище (аналог Яндекс Диска)
+  Примерные кейсы
	- Пользователь регистрируется и получает личное хранилище с квотой (например 10 ГБ)
	- Загружает файлы - маленькие целиком, большие по частям (chunked upload)
	- Организует файлы в папки, переименовывает, перемещает, удаляет в корзину
	- Шарит файл или папку по ссылке с правами: только просмотр или редактирование
	- Скачивает файл - сервер стримит байты не загружая всё в память
	- Восстанавливает предыдущую версию файла если случайно перезаписал

---
 + Основные сложности
	- Загрузка по чанкам, загружает параллельно, сервер собирает обратно. Нужно хранить состояние загрузки и уметь возобновлять после обрыва
	- Нельзя читать файл целиком в память, всё через `io.Reader` / `io.Writer`,
	- Если два пользователя загружают один и тот же файл, физически хранить его нужно один раз.
	- Рекурсивные запросы, подсчёт размера папки включая вложенные, запрет циклических ссылок
	- У каждого объекта своя ACL, нужно проверять права на каждой операции включая вложенные файлы при шаринге папки
	- Атомарно проверить и занять место при загрузке, освободить при удалении

---
+ Примерный стек
	- SQL БД 
	- S3 хранилище
	- Cache 
	- Очередь
	- JWT
	- Docker Compose




## №3 Мониторинг цен

+ Примерные кейсы
	- Пользователь добавляет ссылку на товар и целевую цену - "хочу купить за X рублей"
	- Сервис регулярно проверяет цену на странице товара и сохраняет историю
	- Когда цена достигла цели - пользователь получает уведомление (email или webhook)
	- Пользователь видит график изменения цены за последние N дней
	- Можно отслеживать несколько товаров одновременно, управлять подписками
---
+ Основные сложности
	- Нельзя дудосить один сайт слишком часто, иначе забанят по IP. Нужен механизм как это обойти
	- Если сайт вернул 429 или 503, нужно подождать и повторить, но не бесконечно , нужно решить как именно и сколько ждать, как обрабатывать такие кейсы
	- Настраиваемый и управляемый воркер пул, обоснование сколько именно надо и как именно 
	- У большинства площадок нет открытого апи чтобы парсить цены, и у каждого магазина своя структура HTML, нужна гибкая система селекторов, также цена может быть `1 990 ₽` а не `1990` - нужна нормализация
	- Задачи на парсинг должны создаваться по расписанию, равномерно распределяться во времени чтобы не было спайков нагрузки
	- Если воркер упал на середине и задача вернулась в очередь, повторная обработка не должна создавать дублей в истории цен
	- Корректный механизм парсинга, не просто разные http.Get c готовой ссылкой, а именно механизм собственного парсера для каждой из площадок, плюс это покрывает обход разных антифрод решений у разных площадок(небольшая подсказка, я бы хотел тут видеть интерфейс парсера и разные реализации)

---
 + Примерный стек
	+  SQL БД  
	+  Cache — кэш последней известной цены, rate limiter по домену
	- Очередь
	- Метрики вне обсервабилити (метрики сколько активных задач на парсинг, сколько неуспешных попыток получения цены, время парсинга по доментам )
	- Трейсы от момента создания задачи до отправки уведомления
	- Docker compose 



## №4 Агрегатор новостей

+  Примерные кейсы
	- Система регулярно обходит список RSS-лент и забирает новые статьи
	- Похожие статьи из разных источников об одном событии - схлопываются в одну карточку
	- Каждая статья автоматически получает теги через AI API (технологии, политика, спорт и т.д.)
	- Пользователь выбирает интересующие теги и источники - видит персональную ленту
	- Лента пагинируется, есть полнотекстовый поиск по заголовкам и содержимому

---
+  Основные сложности
	- Одну новость публикуют 10 изданий. Не нужно дублировать её 10 раз, нужно заложить механизм как это делать. (Хеши или какой-то другой?)
	-  API нейросети упал или отвечает медленно. Нужен механизм как на нем не виснуть или например переключиться на другую , здесь же rate limiting к апи и оптимизация траты токенов, выбор достаточной модели
	- Полнотекстовый поиск  в PostgreSQL, правильные индексы
	- Оптимизировать хранение новостей, не нужно хранить старые, автоудаление

---
+ Примерный стек
	- SQL БД  
	- Очередь ( какая удобнее и лучше?)
	- AI API
	- Метрики на состояние очереди тегирования, процент дедупликации
	- Docker Compose
